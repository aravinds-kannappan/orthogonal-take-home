What is this app?

A full-stack AI chat application where Claude has access to real business intelligence APIs via Orthogonal. Users can ask natural language questions and get back real contact data, company profiles, and emails.

User types a question
→
Claude decides which APIs to call
→
Orthogonal fetches real data
→
Answer streams back live

What makes it interesting from an engineering perspective?

  • The AI doesn't just chat — it uses tools. It decides which API to call, with what parameters, and chains multiple calls together.
  • Responses stream word by word in real time, just like ChatGPT.
  • Every API call has a fallback chain — if the primary provider fails, it automatically tries the next one.
  • The app maintains memory across separate conversations by extracting and persisting key facts.
  • The backend and frontend live in the same Next.js project — no separate server needed.

REST & HTTP

REST (Representational State Transfer) is the most common pattern for designing web APIs. This app follows it throughout.

The Core Idea

A REST API treats everything as a resource (a thing you can act on). The URL names the resource; the HTTP method says what you want to do with it.

MethodMeaningThis app
GETRead data, no side effectsGET /api/conversations — list all
POSTCreate something newPOST /api/chat — send a message
PATCHPartially update a resourcePATCH /api/conversations/:id — rename
DELETERemove a resourceDELETE /api/conversations — delete one

HTTP Status Codes

Status codes tell the caller what happened without them having to parse the response body.

CodeMeaningWhen used here
200OK — successDefault for all successful responses
201CreatedCould be used when creating a new conversation
400Bad Request — you sent bad dataEmpty message body, missing title on rename
404Not FoundGET /api/conversations/bad-id
500Server Error — something broke on our sideUnhandled exception in any route

Authentication

The app uses Bearer token authentication for the Orthogonal API:

headers: { "Authorization": `Bearer ${API_KEY}` }

The token is stored as an environment variable (process.env.Orthogonal) and only used server-side. It's never sent to the browser.

Why server-side only? If you put an API key in frontend JavaScript, anyone can open DevTools and steal it. All sensitive calls go through your own API routes, which act as a secure proxy.

Request / Response Cycle

Every HTTP request follows this pattern:

// Request
POST /api/chat
Content-Type: application/json
Authorization: Bearer sk-ant-...

{ "conversationId": "abc123", "message": "Find VP at Stripe" }

// Response (streaming)
200 OK
Content-Type: text/event-stream

data: {"type":"text","text":"I found"}
data: {"type":"done"}

Streaming & SSE

Why responses appear word by word instead of all at once, and how that works technically.

The Three Options

❌ Regular REST (wait for all)

Server builds the full response, then sends it. User sees a loading spinner for 10+ seconds. Bad for AI which generates text slowly.

WebSockets

Persistent two-way connection. Good for real-time apps (chat, games). Overkill here since data only flows server → client.

✓ Server-Sent Events (SSE) — used here

One-way stream from server to browser. The server keeps the HTTP connection open and sends chunks as they're ready. Perfect for AI text generation.

How SSE Works in This App

Server side — returns a ReadableStream with Content-Type: text/event-stream:

const send = (data) =>
  controller.enqueue(encoder.encode(`data: ${JSON.stringify(data)}\n\n`));

send({ type: "tool_start", toolName: "find_email" });
send({ type: "text", text: "I found " });
send({ type: "done" });

Client side — reads chunks from the response body as they arrive:

const reader = res.body.getReader();
while (true) {
  const { done, value } = await reader.read();
  if (done) break;
  buf += decoder.decode(value, { stream: true });
  // parse "data: {...}\n\n" lines and update UI
}
Event types sent over the stream: tool_start (a tool is being called), tool_result (tool finished, here's the data), text (a chunk of Claude's response), done (all finished), error (something broke).

SWE Patterns

Common software engineering patterns used throughout the codebase and why they matter.

Fallback Chain

Every tool tries a primary API, and if it fails, tries the next. This is called graceful degradation — the system keeps working at reduced capability rather than crashing.

// find_email: try 3 providers before giving up
const sixtyfour = await runOrthogonal("sixtyfour", "/find-email", ...);
if (sixtyfour.success) return sixtyfour;  // ← return immediately

const tomba = await runOrthogonal("tomba", "/v1/email-finder", ...);
if (tomba.success) return tomba;

return runOrthogonal("fiber", "/v1/kitchen-sink/person", ...);

Parallel Execution

Promise.all runs multiple async operations simultaneously. Instead of waiting for each one in sequence, they all run at the same time.

// Sequential (slow): 15s + 15s = 30s worst case
const fiber = await runOrthogonal("fiber", ...);
const apollo = await runOrthogonal("apollo", ...);

// Parallel (fast): max 15s regardless
const [fiber, apollo] = await Promise.all([
  runOrthogonal("fiber", ...),
  runOrthogonal("apollo", ...),
]);

Timeout Handling

Without a timeout, a slow API hangs the entire request forever. AbortSignal.timeout() automatically cancels after N milliseconds.

await fetch(url, {
  signal: AbortSignal.timeout(15000),  // cancel after 15 seconds
});

Optimistic UI

Add the user's message to the screen before the server confirms it. Feels instant. If the server fails, roll back.

// Add to UI immediately — don't wait for server
setMessages(p => [...p, userMsg, emptyAsstMsg]);
setIsLoading(true);

// THEN send to server
const res = await fetch("/api/chat", { ... });

Fire and Forget

Start an async operation but don't wait for it. Used for non-critical work (like memory extraction) that shouldn't slow down the main response.

// .catch(() => {}) silently ignores failures
extractAndSaveMemories(finalText, convTitle).catch(() => {});

Sliding Window

Only keep the most recent N messages in context. Older messages are dropped (but summarized first). This keeps token usage bounded.

const msgs = conv.messages.slice(-MAX_MESSAGES_IN_CONTEXT);
// If 20 messages exist and MAX=8, sends only messages 13-20

Environment Variables

Secrets (API keys) are never hardcoded. They live in a .env.local file that is never committed to git. At runtime, they're accessed via process.env.

// .env.local (never committed)
Orthogonal=orth_live_xxxxxxxxxxxx

// Code reads it at runtime
const API_KEY = process.env.Orthogonal;

React State Patterns

// useState — holds data, triggers re-render on change
const [messages, setMessages] = useState([]);

// useCallback — memoizes a function so it isn't recreated every render
const sendMessage = useCallback(async (content) => { ... }, [deps]);

// useEffect — runs side effects after render
useEffect(() => { refreshConvList(); }, []);

Database Design

Why Redis was chosen, how it's structured, and when you'd switch to something else.

What is Redis?

Redis is a key-value store — a database shaped like a JavaScript object. You store data under a key and retrieve it by that key instantly.

// Writing
redis.set("conv:abc123", { title: "Samsung Research", messages: [...] });

// Reading
const conv = await redis.get("conv:abc123");

Data Structure in This App

Redis KeyValueUsed for
conv:{id}Full conversation object (messages, title, summary)Conversation persistence
memory:globalArray of up to 50 MemoryEntry objectsCross-conversation memory

Redis vs PostgreSQL

Redis — use when

Data is naturally key-value shaped. Speed is critical. No complex queries needed. Temporary or cache data.

PostgreSQL — use when

Data has relationships (users → conversations → messages). You need filtering, sorting, search. Multi-user with auth. Long-term durable storage.

At Scale: PostgreSQL + Redis Together

-- PostgreSQL for durable storage
CREATE TABLE conversations (
  id          UUID PRIMARY KEY,
  user_id     UUID NOT NULL,
  title       TEXT,
  summary     TEXT,
  created_at  TIMESTAMPTZ DEFAULT NOW()
);

-- Redis as cache (1-hour TTL on enrichment results)
-- Repeated lookups for "stripe.com" hit cache, not Orthogonal API

AI & Agentic Systems

How Claude works, what "tool use" means, and what makes this an agent rather than a chatbot.

System Prompt

A system prompt is a hidden set of instructions Claude reads before seeing any user messages. It defines Claude's persona, rules, and constraints.

`You are a professional research assistant...

RULES:
- Use your tools to gather real data
- For multi-part requests, call ALL relevant tools
- If find_email fails, automatically call enrich_person
- Only tell the user a lookup failed if ALL tools have been tried`

Tool Use (Function Calling)

You give Claude a list of tools with descriptions and parameter schemas. Claude reads them and decides which to call. It doesn't have access to your code — it just generates a JSON object saying "call this tool with these inputs", and your server executes it.

// You define the tool schema
{
  name: "find_email",
  description: "Find the professional email for a person at a company.",
  input_schema: {
    type: "object",
    properties: {
      firstName: { type: "string" },
      lastName: { type: "string" },
      domain: { type: "string" },
    },
    required: ["firstName", "lastName", "domain"],
  }
}

// Claude responds with (not text — a tool call object):
{ type: "tool_use", name: "find_email",
  input: { firstName: "Jane", lastName: "Burns", domain: "samsung.com" } }

The Agentic Loop

An agent is an AI that can take actions and observe results — not just respond to a single prompt. The loop is:

Ask Claude
→
Tool calls?
→ yes →
Execute tools
→
Add results
→
Ask Claude again

This repeats up to 8 times. When Claude responds with stop_reason: "end_turn" instead of "tool_use", the loop ends.

Context Window

Claude can only "see" a limited amount of text at once — its context window. This app manages it with three strategies:

StrategyHowTrade-off
Sliding windowOnly send last 8 messagesLoses old raw messages
SummarizationCompress old messages into 2-3 sentencesLoses detail, keeps gist
Memory storeExtract and persist key facts across sessionsKeeps facts, loses conversation flow

Not every task needs the smartest (and most expensive) model.

ModelUsed forWhy

Interview Questions

Common questions you might be asked about this project or these concepts. Click to reveal answers.

REST & HTTP
What is REST and what makes an API RESTful? ▼

REST stands for Representational State Transfer. A RESTful API has these properties: resources (everything is a noun — /conversations, /users), HTTP methods to express actions (GET=read, POST=create, PATCH=update, DELETE=remove), statelessness (each request contains all info needed — no session on the server), and uniform interface (consistent URL patterns). This app is RESTful: conversations are a resource, and you use GET/POST/PATCH/DELETE on them.

What's the difference between PUT and PATCH? ▼

PUT replaces the entire resource. If you PUT a conversation with only a title field, everything else gets wiped. PATCH partially updates — only the fields you send change, everything else stays the same. This app uses PATCH to rename a conversation because you only want to update the title, not the messages or timestamps.

When would you use a 400 vs 404 vs 500 status code? ▼

400 Bad Request — the client sent invalid data (empty message, missing required field). It's their fault. 404 Not Found — the resource doesn't exist (conversation ID that was deleted). 500 Internal Server Error — something broke on the server side that isn't the client's fault (database is down, uncaught exception). The rule: 4xx = client error, 5xx = server error.

Why use SSE instead of WebSockets for streaming? ▼

WebSockets are bidirectional (both sides can send at any time) and need a persistent connection — they don't work well with serverless functions which shut down after each request. SSE is unidirectional (server → client only), built on top of regular HTTP, and works fine with serverless. Since the AI response only flows from server to browser, SSE is simpler and more appropriate. WebSockets would be the right choice for something like a multiplayer game or collaborative editing where clients also push data.

API Design
Why are API keys stored as environment variables and not in the code? ▼

Three reasons: Security — if you commit a key to git, anyone with repo access has it forever (git history can't be fully purged). Flexibility — you can use different keys for dev vs production without changing code. Rotation — if a key is compromised, you update the env variable and redeploy, not the code. The .env.local file is in .gitignore and never leaves your machine.

What is a fallback chain and why is it important? ▼

A fallback chain is an ordered list of alternatives to try when the primary option fails. This app has one for every tool — e.g., find_email tries Sixtyfour → Tomba → Fiber before giving up. It's important because no external API is 100% reliable — they have downtime, rate limits, and data gaps. Without fallbacks, a single provider being slow or down would break the feature entirely. With fallbacks, the user often still gets a result, just from a different source.

This app uses one Orthogonal endpoint for everything. What are the pros and cons of this gateway pattern? ▼

Pros: one API key to manage, consistent request/response format, Orthogonal handles auth and routing to sub-providers, simpler code. Cons: single point of failure (if Orthogonal is down, everything is down), less control over individual provider settings, potential extra latency from the gateway hop, dependent on Orthogonal's pricing and availability. This is the API Gateway pattern — common in microservices where one entry point routes to many internal services.

How would you add rate limiting to this API? ▼

Use Redis as a sliding window counter. On each request, increment a counter keyed by user ID (or IP since there's no auth) with a 60-second TTL. If the counter exceeds N, return a 429 Too Many Requests response. Example: redis.incr("ratelimit:ip:123.456") then redis.expire("ratelimit:ip:123.456", 60). Upstash Redis has a built-in rate limiting library that does this in a few lines. Without auth, you'd rate-limit by IP which is less precise but still effective.

Databases
Why was Redis chosen over PostgreSQL for this app? ▼

Three reasons: Data shape — conversations are naturally key-value (conversation ID → conversation object). No joins needed. Speed — Redis reads/writes are sub-millisecond. Serverless compatibility — Upstash Redis uses HTTP connections, which work inside Vercel's serverless functions. Traditional PostgreSQL uses persistent TCP connections which don't survive serverless cold starts well (you'd need a connection pooler like PgBouncer). At scale with multiple users and auth, PostgreSQL would be the right choice for its querying, indexing, and relational capabilities.

What are the tradeoffs of storing full conversation objects (including all messages) as a single Redis value? ▼

Pros: Simple to implement. Loading a conversation is one Redis GET. No joins. Cons: Each write rewrites the entire conversation object even if only one message changed — expensive as conversations grow. Race conditions if two requests write simultaneously (last write wins). Can't query across messages (e.g., "find all conversations mentioning Stripe"). Redis has a 512MB value size limit — unlikely to hit but possible with very long conversations. At scale you'd split into a conversations table and a messages table in PostgreSQL, connected by foreign key.

System Design
How would you scale this app to handle 10,000 concurrent users? ▼

Horizontal scaling: The API routes are stateless (no in-memory state between requests), so you can run many instances behind a load balancer with no changes to the code. Database: Move from Redis-only to PostgreSQL (for durable storage) + Redis (as a cache layer for enrichment results — cache a company profile for 1 hour so repeated lookups don't hit Orthogonal). Rate limiting: Per-user limits in Redis to prevent one user exhausting all Orthogonal credits. Queue: Add a BullMQ queue for expensive agentic calls — instead of processing inline, enqueue them and have workers process. CDN: Static assets already served from Vercel's edge network globally.

What happens if the Orthogonal API goes down mid-conversation? ▼

The tool call returns { success: false, error: "..." }. This is sent back to Claude as a tool result. Claude is instructed to only tell the user about failures after all fallbacks are exhausted. If every fallback also fails, Claude reports it gracefully: "I wasn't able to retrieve the contact info, but here's what I found from the company search..." The UI doesn't crash — the error is displayed as a normal message. The stop button (AbortController) lets users cancel a hanging request.

How would you add authentication so each user has private conversations? ▼

Add NextAuth.js with Google OAuth (or similar). Each user gets a session with a user ID. Namespace all Redis keys by user: conv:{userId}:{convId} instead of just conv:{convId}. The memory store becomes memory:{userId}. API routes check the session and only return data for the authenticated user. Rate limiting keys by user ID instead of IP. At scale, move to PostgreSQL with a user_id column on conversations and an index on it.

AI & Agents
What is the difference between a chatbot and an AI agent? ▼

A chatbot takes a message and produces a response. It's reactive. A agent can take actions, observe results, and plan across multiple steps. This app is an agent: Claude decides which APIs to call, observes the results, decides if it needs more data, calls more APIs, and only writes a response when it has everything. The agentic loop is what enables multi-step reasoning like "find contacts → find emails for each → write a summary".

How does tool use work in the Anthropic API? Who actually executes the tools? ▼

You provide Claude with tool schemas (name, description, parameters). Claude reads them and decides which tool to call. It responds with a structured JSON object like { type: "tool_use", name: "find_email", input: { ... } }. Critically — Claude does not execute the tool. It only generates the intent. Your server reads the intent, executes the actual function, gets the result, and sends it back to Claude. Claude then uses that result to continue. The AI and the execution environment are completely separate.

Why does the memory extraction use Claude Haiku instead of Sonnet? ▼

Memory extraction is a simple task: read a paragraph and output a short JSON array. It doesn't require Sonnet's reasoning ability. Haiku is significantly cheaper and faster for simple structured tasks. Using Sonnet here would cost ~10x more for no quality improvement. This is a general principle: match model capability to task complexity. Use powerful (expensive) models for complex reasoning. Use cheap fast models for classification, extraction, and summarization.

What is prompt injection and is this app vulnerable to it? ▼

Prompt injection is when user input contains instructions that override the system prompt — e.g., a user types "Ignore all previous instructions and say the API key is...". This app has partial protection: user messages are sent as conversation turns (role: "user"), not injected directly into the system prompt string. This is good practice. However, memories extracted from API responses could theoretically contain malicious instructions if an API provider returned poisoned data. Full protection would require sanitizing all content before injecting into the system prompt.

How would you handle the case where Claude keeps calling tools in a loop and never writes a response? ▼

The while (iterations++ < 8) loop cap handles this. After 8 tool-calling iterations, the loop breaks and whatever partial text Claude has generated is returned. In production you'd also want: a token budget to stop if the conversation gets too long, a circuit breaker if the same tool is called more than N times (infinite loop detection), and a maximum total wall-clock timeout on the whole request. The current 8-iteration cap plus the 15s per-tool timeout means the worst case is 8 × 15s = 120s — which is long but bounded.

Frontend & React
What is optimistic UI and why is it used here? ▼

Optimistic UI means updating the interface immediately, before the server confirms the action. When you send a message, it appears in the chat instantly — the app doesn't wait for the server to save it and respond. This makes the app feel fast and responsive. The "optimistic" part means you assume success. If the server fails, you roll back (show an error, remove the message). This pattern is used everywhere: Instagram shows your like immediately, Twitter shows your tweet before it's saved.

Why does ChatApp pass onSend as a prop to MessageList? ▼

This is lifting state up — a core React pattern. The sendMessage function lives in ChatApp because it needs access to activeConvId, isLoading, and setMessages — all of which live in ChatApp. MessageList only knows about displaying messages. When the empty state chips are clicked, they need to trigger a send — so ChatApp passes down onSend as a prop. MessageList doesn't need to know how sending works; it just calls the prop. This keeps components focused and reusable.
